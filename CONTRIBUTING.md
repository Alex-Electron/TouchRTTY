# Правила работы с репозиторием TouchRTTY (SOP)

Этот документ описывает стандартные процедуры (Standard Operating Procedure) по ведению репозитория, выпуску релизов и интеграции нового кода. Проект использует адаптированный **GitHub Flow** с элементами Trunk-Based Development.

## 1. Архитектура веток (Branching Strategy)

### `main` (Главная ветка)
* **Статус:** Святая святых. Здесь лежит ТОЛЬКО стабильный, протестированный на железе код.
* **Правило:** Прямые коммиты в `main` запрещены (или крайне нежелательны). Любой код попадает сюда только через слияние (Merge) из другой ветки.
* **Состояние:** Код в `main` всегда должен успешно компилироваться в рабочий `.uf2` файл и запускаться на плате без критических сбоев.

### `feature/*` (Ветки разработки)
* **Статус:** Рабочие полигоны.
* **Именование:** `feature/название-фичи` или `fix/описание-бага`.
* **Примеры:** `feature/fractional-timing`, `feature/cw-fast-mode`, `fix/ili9488-red-bug`.
* **Правило:** Создаются от актуального `main`. Когда фича готова и оттестирована на макетной плате, ветка сливается в `main` (через Pull Request или локальный `git merge`) и удаляется.

## 2. Версионирование и Релизы (Releases)

Формат версий привязан к номерам успешных сборок (Builds). Например, билд 172 конвертируется в релиз `v1.72` (где 1 — глобальная версия/архитектура, а 72 — номер удачного билда). 

### Пайплайн выпуска релиза:
1. Вы завершили тестирование фичи в ветке `feature/new-dsp`.
2. Сливаете её в `main`.
3. В Git создаете тег с названием версии: 
   ```bash
   git tag v1.72
   ```
4. Отправляете тег на сервер GitHub: 
   ```bash
   git push origin v1.72
   ```
5. В интерфейсе GitHub переходите в раздел **Releases -> Draft a new release**.
6. **Что прикреплять к релизу:**
   * **Краткий ченджлог (Changelog):** что добавили, что починили (например: "Добавлен логарифмический Squelch, исправлен баг красного водопада").
   * **Бинарники (Assets):** Обязательно прикрепляйте готовый файл `rtty_decoder.uf2` из папки `build`. Это позволит другим радиолюбителям просто скачать файл и закинуть его на Pico 2 как на флешку, минуя процесс установки CMake и Pico SDK.

## 3. Структура репозитория (Repository Layout)

Проект организован по стандарту C/C++ проектов для встроенных систем:

```text
/
├── .gitignore          # Исключения для Git
├── CMakeLists.txt      # Главный файл сборки (конфигурация CMake)
├── pico_sdk_import.cmake # Скрипт подтяжки Raspberry Pi Pico SDK
├── README.md           # Главное описание проекта, фото интерфейса
├── CONTRIBUTING.md     # Правила работы с репозиторием (этот файл)
├── /docs               # Архитектурные планы и итоги фаз (.md)
├── /lib                # Внешние зависимости (LovyanGFX, FatFs)
├── /src                # Исходные коды (C++)
│   ├── main.cpp        # Главный цикл и оркестрация (Core 0 / Core 1)
│   ├── /dsp            # Математика демодулятора (FFT, DPLL, Фильтры)
│   ├── /ui             # Отрисовка интерфейса
│   └── /display        # DMA драйвер экрана (PIO/SPI)
└── /tools              # Вспомогательные Python скрипты и симулятор
```
