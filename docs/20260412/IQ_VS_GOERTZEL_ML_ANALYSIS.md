# Анализ: I/Q + DPLL vs Goertzel + Character-Level ML

**Дата:** 2026-04-12
**Контекст:** TouchRTTY build g259, RP2350 Cortex-M33 @ 300-500 МГц + 8 МБ PSRAM

---

## 1. Текущий алгоритм: I/Q когерентный + Biquad + DPLL

### Плюсы

1. **Непрерывная обработка** — каждый сэмпл обрабатывается моментально, нет блочной задержки. Результат доступен мгновенно.

2. **Плавный waterfall/спектр** — I/Q демодулятор даёт непрерывный поток данных для визуализации в реальном времени. Waterfall 20-22 FPS.

3. **Гибкость ATC** — отдельные envelope-трекеры для Mark и Space с настраиваемыми attack/release. Можно тюнить под конкретные условия.

4. **DPLL отслеживает drift бода** — PI-контроллер компенсирует рассогласование ±5% между передатчиком и приёмником непрерывно, не только в моменты переходов.

5. **Хорошая работа при переменном бод** — непрерывный DPLL адаптируется к "плавающему" боду (ручная передача, нестабильные генераторы).

6. **Простая отладка** — каждый этап конвейера можно визуализировать независимо: огибающая Mark, огибающая Space, фаза DPLL, дискриминатор.

7. **Уже работает и отлажен** — build g259, Phase 3 завершена, проверен на реальных сигналах (WebSDR Twente, DWD).

8. **Независимость от baud rate** — при смене бод достаточно пересчитать biquad коэффициенты и DPLL rate. Структура не меняется.

### Минусы

1. **Biquad LPF ≠ matched filter** — biquad Butterworth Q=0.707 не является оптимальным фильтром для тонального burst. Теряет ~1-2 дБ SNR по сравнению с matched filter.

2. **28 МАС/сэмпл** — 4 biquad (по 5 МАС) + NCO + LUT lookup + AGC. В 7× тяжелее Goertzel.

3. **Zero-crossing DPLL обновляется только на переходах** — при длинных сериях Mark или Space (нет переходов) DPLL "слепнет" и может дрейфовать.

4. **Integrate-and-dump = равновесное усреднение** — все сэмплы в символе имеют одинаковый вес. Сэмплы на границах (с ISI) вносят столько же, сколько центральные (чистые). Субоптимально.

5. **Побитовое решение** — каждый бит решается независимо. Не используется известная структура Baudot-фрейма (start=0, stop=1). Framing error обнаруживается только после декодирования.

6. **Нет soft-output** — решение жёсткое (integrate > 0 → Mark, иначе Space). Нет метрики уверенности, нельзя использовать для коррекции или вероятностного декодирования.

7. **ATC требует expf()** — текущая реализация вызывает expf() на каждом сэмпле (исправляемо кэшированием, но это дефект реализации).

## 2. Goertzel + Character-Level ML

### Плюсы

1. **Matched filter = теоретически оптимальный** — Goertzel за 1 символ = корреляция с ожидаемым тональным burst. Максимизирует SNR на выходе по теореме Неймана-Пирсона. **+1-2 дБ** vs biquad LPF.

2. **Character-level ML использует избыточность Baudot** — start(0) + stop(1) = 2.5 известных бита из 7.5. Это 33% избыточности, "бесплатно" встроенной в протокол. **+2-3 дБ** сверх побитового.

3. **Framing errors практически исключены** — принимаются только валидные 7.5-битные паттерны Baudot. Невалидные комбинации отсеиваются самим алгоритмом.

4. **Синхронизация = побочный продукт** — фаза с максимальной корреляцией автоматически даёт оптимальный timing. Не нужен отдельный DPLL.

5. **Всего 4 МАС/сэмпл** (базовый Goertzel) — в 7× эффективнее текущей реализации. Освобождает CPU для шумоподавления, FT8, DRM.

6. **Character ML: <1% CPU на RP2350** — 256 корреляций × 2200 сэмплов = 1.9 мс из 220 мс доступных (при 45 бод). Огромный запас.

7. **Soft output** — метрика корреляции для каждого символа = мера уверенности. Можно использовать для Байесовской коррекции, отображения "сомнительных" символов другим цветом на экране.

8. **Порог декодирования ~−13 дБ SNR** — на уровне лучших в мире (2Tone). Текущий: ~−7 дБ. Разница **6 дБ** — это как усилить антенну в 4 раза.

9. **Идеально для мульти-фазной синхронизации (MMTTY-style)** — 8 фаз × 2 тона = 32 МАС/сэмпл. Заменяет DPLL + zero-crossing целиком.

### Минусы

1. **Блочная задержка** — решение принимается только в конце символа (7.5 бит). При 45 бод: задержка ~165 мс. При 100 бод: ~75 мс. Для RTTY это приемлемо (текст), но не подходит для real-time аудио.

2. **Нет непрерывного waterfall с демодулятора** — Goertzel выдаёт мощность раз в блок, не на каждый сэмпл. Для waterfall нужен отдельный FFT (который у нас уже есть на Core 1). Waterfall продолжит работать, но от FFT, не от демодулятора.

3. **Чувствительность к ошибке частоты** — длинное окно корреляции (7.5 бит = ~165 мс при 45 бод) усиливает влияние частотной ошибки. При смещении >5 Гц корреляция деградирует. Нужна хорошая AFC.

4. **Чувствительность к ошибке бода** — шаблоны генерируются для точного baud rate. Рассогласование >1-2% вызывает рассинхронизацию корреляции. Текущий DPLL с PI-контроллером компенсирует ±5%.

5. **Сложнее отладка** — нет промежуточных I/Q огибающих для визуализации. "Чёрный ящик": на входе — аудио, на выходе — символ. Для tuning indicator нужны дополнительные вычисления.

6. **Переходные процессы при смене символов** — корреляция начинается с начала блока; если начальная фаза не совпадает с реальным символом, первые 1-2 символа могут быть потеряны. Нужна стадия "захвата".

7. **Не подходит для CW/PSK** — специфичен для Baudot RTTY. Для CW нужна другая архитектура. Текущий I/Q-подход более универсален для мультирежимного приёмника.

8. **Сложность реализации** — character-level ML требует предвычисления 32 × 8 шаблонов для каждой комбинации baud/shift/stop. При смене параметров — пересчёт шаблонов.

## 3. Сравнительная таблица

| Критерий | Текущий (I/Q + DPLL) | Goertzel + ML |
|---|---|---|
| **Порог SNR** | ~−7 дБ | **~−13 дБ (+6 дБ!)** |
| **CPU/сэмпл** | ~28 МАС | **~4 МАС** (в 7× меньше) |
| **CPU на character ML** | — | ~1% Core 0 дополнительно |
| **Задержка** | **~0 мс** (per-sample) | ~165 мс (per-character @ 45 бод) |
| **Оптимальность фильтра** | Субоптимальный (biquad) | **Оптимальный (matched)** |
| **Синхронизация** | DPLL (слепнет без переходов) | **Multi-phase (всегда работает)** |
| **Framing errors** | Обнаруживаются после факта | **Исключены алгоритмически** |
| **Soft output** | Нет | **Да (метрика корреляции)** |
| **Устойчивость к drift бода** | **±5% (DPLL компенсирует)** | ±1-2% (фиксированный шаблон) |
| **Устойчивость к drift частоты** | **±100 Гц (AFC)** | ±5 Гц (длинная корреляция) |
| **Waterfall от демодулятора** | **Непрерывный** | Нет (нужен отдельный FFT) |
| **Визуализация tuning** | **I/Q огибающие, Лиссажу** | Нет (чёрный ящик) |
| **Универсальность** | **CW, PSK, RTTY** | Только RTTY Baudot |
| **Сложность кода** | Средняя (есть, работает) | Высокая (переписывать) |
| **Отладка** | **Прозрачная** (все этапы видны) | Сложная |
| **Нужен отдельный AFC** | Да (FFT peak) | Да (более точный) |
| **Компенсация селективного фединга** | **ATC (FASR envelope)** | Неявная (ML усредняет) |

## 4. Решение: гибридная архитектура

Не заменять, а **добавить**. Оба алгоритма работают параллельно, каждый делает то, в чём он лучше.

### 4.1. Архитектура

```
                    ┌─ I/Q + DPLL ─→ Waterfall, Lissajous, tuning indicators
ADC → FIR → AGC ───┤                  Mark/Space envelopes для UI
                    │                  Fallback-декодер (drift >2%, freq >5 Гц)
                    │
                    └─ Goertzel ML ─→ Основной выход символов RTTY
                                      Soft-decision метрики
                                      Confidence per character
```

### 4.2. Разделение обязанностей

| Функция | Кто выполняет | Почему |
|---|---|---|
| **Декодирование RTTY** | Goertzel ML | +6 дБ лучше, optimal |
| **Waterfall** | FFT (Core 1, как сейчас) | Уже работает |
| **Спектр** | FFT (Core 1) | Уже работает |
| **Лиссажу (XY scope)** | I/Q pipeline | Нужны непрерывные I/Q |
| **Tuning indicator** | I/Q pipeline | Mark/Space envelopes |
| **AFC** | FFT (Core 1) + sub-bin interpolation | Питает оба алгоритма |
| **Сквелч** | FFT-based SNR (Core 1) | Уже работает |
| **Fallback при drift** | I/Q + DPLL | Толерантнее к ±5% бод, ±100 Гц |
| **Confidence display** | Goertzel ML soft output | Цветовая метрика символов |

### 4.3. Автоматическое переключение

```cpp
// Goertzel ML — основной декодер
char ml_char = goertzel_ml_decode(audio_block);
float ml_confidence = goertzel_ml_confidence();

// I/Q DPLL — fallback декодер (работает параллельно)
char iq_char = iq_dpll_decode(sample);  // per-sample, как сейчас

// Логика выбора:
if (ml_confidence > 0.6f) {
    // ML уверен — использовать ML (оптимальный)
    output_char = ml_char;
    decoder_mode = DECODER_ML;
} else if (freq_error > 5.0f || baud_error > 2.0f) {
    // Большой drift — ML деградирует, I/Q справляется лучше
    output_char = iq_char;
    decoder_mode = DECODER_IQ_FALLBACK;
} else {
    // Низкая уверенность, но drift малый — попробовать ML
    // Показать символ жёлтым цветом (uncertain)
    output_char = ml_char;
    decoder_mode = DECODER_ML_UNCERTAIN;
}
```

### 4.4. Визуализация уверенности на экране

```
┌──────────────────────────────────┐
│ RTTY 45.45/170  ML  SNR:12 dB   │
├──────────────────────────────────┤
│                                  │
│ CQ CQ CQ DE RA3AAA RA3AAA K     │  ← зелёный = ML confidence >80%
│ RA3AAA DE DL2KL DL2KL 599 599   │
│ TU 73 DE DL2KL SK               │
│ CQ CQ CQ DE W1AW  W?AW  K      │  ← "?" жёлтый = ML confidence 40-60%
│ R█ 3A□A DE W1AW 599 001 BK      │  ← "□" красный = fallback на I/Q
│                                  │
├──────────────────────────────────┤
│ [ML:94%] [IQ:OK] [AFC:+3Hz]     │
└──────────────────────────────────┘
```

- **Зелёный** текст: ML confidence >80% — максимальная надёжность
- **Жёлтый** текст: ML confidence 40-80% — сигнал на пределе
- **Красный** текст: fallback на I/Q — drift или потеря ML lock

### 4.5. Бюджет CPU (Core 0)

| Компонент | CPU @ 300 МГц | CPU @ 500 МГц |
|---|---|---|
| FIR 63-tap (symmetric → 32 MAC) | ~1.5% | ~0.9% |
| AGC | ~0.3% | ~0.2% |
| I/Q + Biquad ×4 (для UI/fallback) | ~3.5% | ~2.1% |
| Goertzel ×2 (базовый) | ~0.2% | ~0.1% |
| Goertzel Multi-phase ×16 | ~0.5% | ~0.3% |
| Character ML (256 корреляций/символ) | ~0.9% | ~0.5% |
| Noise blanker | ~0.1% | ~0.1% |
| Spectral subtraction | ~0.5% | ~0.3% |
| **Итого** | **~7.5%** | **~4.5%** |
| **Запас для CW/FT8/DRM** | **~92%** | **~95%** |

Оба алгоритма одновременно + шумоподавление = всего 7.5% Core 0 @ 300 МГц. Огромный запас.

### 4.6. Порядок внедрения

```
Этап 1 (1-2 дня): Добавить базовый Goertzel ×2 параллельно I/Q
  ├── Goertzel Mark + Goertzel Space
  ├── Простое побитовое решение (как сейчас, но с matched filter)
  ├── Сравнить BER с I/Q pipeline на тех же сигналах
  └── Ожидаемый выигрыш: +1-2 дБ

Этап 2 (2-3 дня): Multi-phase Goertzel (MMTTY-style)
  ├── 8 фаз × 2 тона = 16 Goertzel
  ├── Заменить DPLL на multi-correlator timing
  ├── Проверить на сигналах с drift бода
  └── Ожидаемый выигрыш: +1 дБ дополнительно

Этап 3 (3-5 дней): Character-level ML (2Tone-style)
  ├── 32 символа × 8 фаз = 256 корреляций
  ├── ML выбор символа + soft confidence
  ├── Интеграция fallback на I/Q
  ├── Цветовая индикация confidence на UI
  └── Ожидаемый выигрыш: +2-3 дБ дополнительно

Этап 4 (1-2 дня): Байесовский prior + тюнинг
  ├── Частотная статистика ITA2 символов
  ├── Контекстная валидация (FIGS/LTRS последовательности)
  ├── Параметрический тюнинг на реальных сигналах
  └── Ожидаемый выигрыш: +0.5-1 дБ
```

### 4.7. Итоговый выигрыш гибридной архитектуры

| Параметр | Только I/Q (сейчас) | Гибридная (I/Q + ML) |
|---|---|---|
| **Порог SNR** | ~−7 дБ | **~−13 дБ** |
| **Framing errors** | Есть | **Практически нет** |
| **Визуализация** | Waterfall + Лиссажу | **Waterfall + Лиссажу + confidence** |
| **Drift бода >2%** | Компенсирует (DPLL) | **Компенсирует (fallback на I/Q)** |
| **Drift частоты >5 Гц** | Компенсирует (AFC) | **Компенсирует (fallback на I/Q)** |
| **CPU Core 0** | 4-6% | **~7.5%** |
| **Soft output** | Нет | **Да** |
| **Позиция среди декодеров** | fldigi/MixW | **2Tone / лучший в мире** |

Гибридная архитектура берёт **лучшее от обоих**: оптимальное декодирование Goertzel ML + визуализация и drift-толерантность I/Q + автоматический fallback.
